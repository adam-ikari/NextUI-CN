# NextUI 模拟器重构失败报告

## 重构目标
实现真实的 NextUI 框架在无头模式下运行并生成 UI 截图，用于 CI/CD 环境的视觉回归测试。

## 重构结果
❌ **失败**

## 核心问题
真实 NextUI 框架过于复杂，不适合用于简单的 CI/CD 截图测试。

### 详细问题分析

### 1. 架构复杂性
- **真实框架依赖**：需要完整的 ROM 文件、模拟器核心、pak 文件等
- **状态管理复杂**：多层状态管理（screen、menu、setting、game_switcher）
- **资源依赖严重**：需要 SDL2_image、SDL2_ttf、字体文件、资源图片等
- **平台特定代码**：大量平台特定的 OpenGL 和图形渲染代码

### 2. 技术障碍

#### 2.1 无头模式兼容性
- OpenGL 在无头环境下表现不稳定
- `GFX_blitHardwareGroup` 函数在无头模式下导致程序挂起
- 纹理和着色器资源在无头模式下无法正确初始化
- 窗口管理代码在无头模式下不兼容

#### 2.2 资源管理
- 字体文件路径依赖复杂的平台配置
- 资源图片需要 SDL2_image 库支持
- 资源加载失败时没有优雅的降级处理
- 资源路径使用硬编码的绝对路径，不适合 CI 环境

#### 2.3 状态管理问题
- `currentScreen`、`menu_key_held`、`show_setting` 等状态变量交织复杂
- 状态转换逻辑分散在多个位置
- 自动遍历需要模拟真实的按键序列，难以精确控制
- 屏幕转换动画和状态同步问题

### 2.4 主循环执行问题
- 主循环在 `GFX_blitHardwareGroup` 调用时停止执行
- 可能是互斥锁死锁、OpenGL 资源访问问题或无限循环
- 无有效的错误处理和恢复机制
- 调试困难，缺少详细的执行跟踪

### 3. CI/CD 集成问题

#### 3.1 环境依赖
- 需要安装 SDL2、SDL2_image、SDL2_ttf 等多个库
- 需要配置 OpenGL 无头渲染
- 需要创建和配置 ROM 文件、pak 文件等测试资源
- 构建和运行时间较长

#### 3.2 维护成本
- 代码修改需要理解复杂的 NextUI 架构
- 每次更改可能影响多个屏幕和状态
- 截图生成逻辑与业务逻辑耦合严重
- 难以进行单元测试和集成测试

### 4. 与重构前的对比

#### 重构前（simulator.c - 简单模拟）
```c
// 优点：
- 代码简洁，易于理解和维护
- 不依赖外部资源（ROM、pak、字体等）
- 独立的 SDL2 渲染，无复杂状态管理
- 可以精确控制截图生成时机和内容
- 生成稳定的 18 张截图用于视觉回归测试

// 缺点：
- 不是真实的 NextUI 框架
- 需要手动同步 UI 变化
```

#### 重构后（nextui.c - 真实框架）
```c
// 优点：
- 使用真实的 NextUI 框架代码
- 测试的是实际运行环境

// 缺点：
- 代码复杂，难以理解和维护
- 依赖大量外部资源
- 状态管理复杂，难以精确控制
- 在无头模式下运行不稳定
- 无法生成完整的截图序列
- 主循环执行阻塞，无法完成自动遍历
```

## 失败的具体表现

### 执行失败
1. 主循环在第一帧后停止执行
2. 程序在 `GFX_blitHardwareGroup` 调用时挂起
3. 自动遍历逻辑无法执行到截图保存步骤
4. 无法生成预期的 18 张截图

### 截图质量失败
1. 即使生成了截图，只显示色块而非完整 UI
2. 缺少字体渲染，所有文本显示为空白
3. 资源图片缺失，UI 元素不完整
4. 与实际 NextUI 界面差异巨大

### CI/CD 集成失败
1. 构建时间长，依赖复杂的库安装
2. 运行不稳定，经常挂起或崩溃
3. 截图结果不一致，无法作为可靠的回归测试基准
4. 维护成本高，每次更改都需要调试复杂的状态问题

## 根本原因分析

### 架构设计缺陷
**错误假设**：真实的 NextUI 框架可以简单地用于 CI/CD 截图测试

**实际情况**：
- NextUI 框架是为实际硬件设计的，不是为自动化测试设计的
- 框架包含大量的平台特定代码和硬件交互逻辑
- 状态管理和渲染逻辑高度耦合，不适合解耦用于测试

### 工具选择错误
**错误假设**：使用真实框架比模拟器更准确

**实际情况**：
- 对于视觉回归测试，**模拟器更合适**
- 真实框架引入了不必要的复杂性和不稳定性
- 模拟器可以提供更稳定、可控的测试环境

## 经验教训

### 1. 工具适用性
- 真实框架 ≠ 自动化测试工具
- 不同的目的需要不同的工具
- 简单的问题应该用简单的解决方案

### 2. 渐进式开发
- 应该先验证基本可行性，再进行全面重构
- 需要在真实环境中测试技术假设
- 保留退路和备选方案

### 3. 复杂度管理
- 不要引入不必要的复杂性
- 每个依赖都会增加失败点
- 可维护性比"真实性"更重要

## 建议

### 短期方案
1. **回退到重构前的 simulator.c 方案**
2. 保持简单、稳定的截图生成功能
3. 使用模拟器作为 CI/CD 的视觉回归测试工具

### 长期方案
1. **分离关注点**：
   - CI/CD 截图测试 → 使用轻量级模拟器
   - 真实设备测试 → 使用真实框架在真实硬件上
   
2. **改进模拟器**：
   - 根据真实的 NextUI 更新模拟器的 UI 布局
   - 添加更多截图场景和覆盖范围
   - 改进自动遍历逻辑

3. **真实框架测试**：
   - 保留真实框架用于实际设备测试
   - 在有头环境下进行手动测试
   - 使用真实的游戏 ROM 和硬件

## 结论

这次重构证明了：**对于 CI/CD 自动化截图测试，模拟器方案远优于真实框架方案**。

真实框架的复杂性和平台依赖性使得它不适合用于自动化测试环境。应该回归到简单、稳定、可控的模拟器方案，这样可以在保证测试效果的同时，大幅降低维护成本和失败风险。

## 后续行动

1. 删除 `refactor/screen-framework` 分支
2. 从 `main` 分支创建新的重构分支
3. 采用渐进式方法，小步快跑
4. 优先考虑稳定性和可维护性
5. 保留现有工作的有用部分（如 CI 配置、测试脚本等）
